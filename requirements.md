# 【要件定義書】ShareKeep（シェアキープ）

## 1. プロジェクト概要

### 1.1 テーマと社会的価値 (CSV)
* **テーマ**: CSV (Creating Shared Value = 共有価値の創造)。社会課題の解決を企業の成長エンジンへ変える。
* **対象とする社会課題**: 宅配便の「再配達」に伴うCO2排出量の増加、および配送業者のコスト・労力逼迫。
* **アプローチ**: 地域の繋がりを活用し、社会課題を「環境貢献度（CO2削減量）」と「ゲーミフィケーション（育成）」という目に見える価値に変換する。

### 1.2 サービスコンセプト
> **「近所でキープ、地球に優しく。」**
> 同じマンションや半径50m以内の近所に住む在宅ワーカーやシニア（代理人）が、不在のユーザー（受取人）の荷物を一時的に代理受領（置き配キープ）する、エコシステム型マッチングプラットフォーム。

### 1.3 ターゲットユーザー
* **受取人**: 宅配ボックスがない住宅（一軒家・小規模アパート）に住み、日中不在がち。近隣にコンビニやPUDOなどの受取スポットが少ないEC利用者。
* **代理人**: 在宅ワーカー、シニア、または地域の小さな個人商店（クリーニング店など）など、日中に対象エリアに滞在している人。
* **配達員**: 大手・中小配送会社のドライバー。

---

## 2. 業務フローとQRシステム

荷物の紛失を防ぎ、ハッカソンのデモで一連の流れを証明するため、2段階の「ワンタイムQRコード認証」によるステータス管理を行います。

### 2.1 配送・受取フロー

- **【発送通知後・事前登録】**
  受取人がECショップから**発送通知（伝票番号記載）を受け取った後**、アプリに「伝票番号(荷物ID)」を登録し、代理受取を申請。
  この時点での荷物ステータスは **「-: 登録済み（配達前）」** とし、配達が試みられるまで待機状態とする。

- **【配達失敗・不在報告・マッチング】**
  配達員が現地で不在を確認 ➔ **配達員がアプリ上の「不在報告」ボタンを押す** ➔ 荷物ステータスが「0: 不在(代理待ち)」に更新 ➔ アプリが「同じ郵便番号」または「同じダミーマンションID」の対応可能な代理人を自動マッチング ➔ マッチング成立と同時に荷物ステータスが **「1: 代理人へ配送中」** に更新（代理人へSupabase Realtimeでプッシュ通知、受取人へも自動通知）

- **【1次引き渡し：配達員 ➔ 代理人】**
  代理人がアプリで「受取用QR」を表示 ➔ 配達員がカメラでスキャン ➔ 荷物ステータスが **「2: 代理キープ中」** に更新（受取人へ自動通知）

- **【2次引き渡し：代理人 ➔ 受取人】**
  受取人が代理人宅へ訪問 ➔ 受取人がアプリで「引渡用QR」を表示 ➔ 代理人がカメラでスキャン ➔ 荷物ステータスが **「3: 引き渡し完了」** に更新（双方にCO2削減量とXPを付与）

## 3. 機能要件 (画面・バックエンド一覧)

ハッカソンの期間内に確実に動作させるため、機能を「必須（MVP）」と「モック（画面のみ/後回し）」に分類しています。

### 3.1 フロントエンド機能

#### ① 共通・認証
* **サインイン / 新規登録画面**: ユーザー（受取人/代理人共通）と「配達員」の権限を切り替えてログイン可能。
* **メイン画面（ホーム）**:
  * アプリの核となる**「育てる何か（木やキャラクター）」のビジュアル表示**。
  * 「受取モード」と「代理モード」を瞬時に切り替えるタブバー。

#### ② 受取人モード画面
* **荷物一覧画面**: 登録した伝票番号と現在のステータスを表示。
* **マッチング・待機画面**: 代理人決定後、「お届け完了までお待ちください」のメッセージを表示。
* **受取準備画面**: 代理人に荷物が届いた後、代理人の情報（場所・名前）を表示。「今から取りに行く」ボタン、および代理人に提示する**引き渡し確認用QRコード**を表示。
* **完了画面**: 「引き渡し完了」のメッセージと、今回の**CO2の推定削減量**を表示。

#### ③ 代理人モード画面
* **代理人設定画面**: 必須（住所、受取可能曜日・時間帯）、任意（緊急連絡先）の入力。
* **受取対応画面（請負リスト）**:
  * **マッチング通知の受信表示**: Supabase Realtimeを通じて新規マッチングをリアルタイムで表示。
  * 配達員に提示する**ワンタイムQRコード表示ボタン**。
  * 受取人が今から来るかのステータス表示。
  * 受取人のQRコードを読み取るための**カメラ起動ボタン**。
* **完了画面**: 受取人同様、完了メッセージと貢献度を表示。

#### ④ 配達員用画面
* **配達リスト・マップ画面**: 不在時に「おすすめの代理人（住所・位置）」をマップおよびリストで提示。（マップ表示にはpickup_spotsの座標を使用）
* **不在報告ボタン**: 不在確認時に押すことでマッチング処理を起動するボタン。
* **スキャン画面**: 代理人のQRコードを読み取るための**カメラ起動画面**。

#### ⑤ 【モック対応】優先度：低
* **ポイント・特典交換画面**: ボタンやUIのみ配置。タップ時は「開発中」のポップアップを表示。
* **トラブル通報ボタン**: 万が一の破損・紛失時の運営への通報窓口（UIのみ）。

---

### 3.2 バックエンド機能 (API & ロジック)

* **ユーザー・プロファイル管理**: ユーザー、配達員のCRUD、および代理人の活動可能時間の保存。
* **簡易マッチングロジック**: ハッカソン用に簡略化。「同じ郵便番号」または「同じダミーマンションID」の代理人を抽出。マッチング成立時にSupabase Realtimeで代理人へ通知を送信し、ステータスを「1: 代理人へ配送中」に更新する。（pickup_spotsの緯度経度はマッチングには使用せず、④配達員マップ画面の表示用途に限定する）
* **QR認証・ステータス更新API (最優先)**:
  * API① (配達員用): 代理人QRの検証 ➔ ステータスを「2: 代理キープ中」に変更。
  * API② (代理人用): 受取人QRの検証 ➔ ステータスを「3: 引き渡し完了」に変更。
* **ゲーミフィケーション・計算ロジック**:
  * 引き渡し完了時、固定の係数を掛け合わせてCO2削減量を算出（例: 再配達1回 = CO2 0.5kg削減）。
  * 完了時にユーザーの**XP（経験値）**をインクリメント。XPは「育てる何か」の成長ステージに反映する育成用の値。
  * **ポイント**はXPとは独立した交換用通貨として付与する（将来の特典交換機能向け。MVP段階では蓄積のみ行い、交換機能はモック対応）。

## 4. 技術構成案

フロントエンドはReact Native (Expo)、バックエンドは極力工数を削減できる**Supabase**を構成案とします。

### 4.1 フロントエンド
* **フレームワーク**: React Native (Expo) / TypeScript
* **ルーティング**: Expo Router または React Navigation
* **UI・グラフィック**: Three.js / React Three Fiber（「育てる何か」の3D表示用。**実装開始から2時間経過しても動作しない場合は即座に2D画像切り替え方式にフォールバックする**）
* **機能ライブラリ**: Expo Camera（QRスキャン用）, QR生成ライブラリ
* **状態管理**: Zustand または React Context API

### 4.2 バックエンド (Supabase推奨)
* **BaaS**: Supabase (認証、PostgreSQL、リアルタイム通知機能を一括担保)
* **データベース設計案 (最低限必要な5テーブル)**:

```sql
-- ユーザー情報
-- users (id, name, role, xp, points);
-- ※ xp: 「育てる何か」の成長に使う経験値。points: 将来の特典交換用通貨（MVP段階では蓄積のみ）

-- 代理スポット・代理人情報
-- pickup_spots (id, user_id, address, postal_code, mansion_id, latitude, longitude, available_time);
-- ※ postal_code / mansion_id: マッチングに使用。latitude / longitude: 配達員マップ表示にのみ使用

-- 荷物・配送管理 (最も重要)
-- deliveries (id, tracking_number, recipient_id, pickup_spot_id, status, co2_saved);
-- ※ statusの定義:
--   -1: 登録済み（配達前・待機中）
--    0: 不在(代理待ち)          ← 配達員が「不在報告」ボタンを押した時点
--    1: 代理人へ配送中          ← マッチング成立時に自動更新
--    2: 代理キープ中            ← API①（配達員QRスキャン）で更新
--    3: 引き渡し完了            ← API②（代理人QRスキャン）で更新

-- QRコード検証用トークン
-- qr_tokens (id, delivery_id, token, purpose, expires_at, used_at);

-- タイムラインログ
-- delivery_events (id, delivery_id, event_type, actor_id, created_at);
```
