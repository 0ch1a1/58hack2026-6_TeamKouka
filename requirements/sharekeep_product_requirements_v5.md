最終更新: 2026-06-13  
対象: 58th Hackathon / Team Kouka / ShareKeep  
位置づけ: 発表・企画・設計判断の正本。開発担当向けの詳細機能一覧は `sharekeep_function_requirements_revised_v2.md` を参照。

## 0. このドキュメントの目的

このドキュメントは、ShareKeep を「近所の誰かに荷物を預けるアプリ」ではなく、**地域の承認済み代理受取スポットを使って再配達を減らすラストワンマイル補完インフラ**として整理するための要件定義である。

添付の開発要件では、認証、荷物管理、QR、マッチング、配達員・受取人・代理人フローなどの基本機能は整理されている。一方で、発表や審査で突っ込まれやすい以下の論点がまだ薄かった。

- 「代理人」ではなく、なぜ「代理受取スポット」として扱うのか
- どのスポットなら安全に荷物を預けられるのか
- 受取人がリアルタイムで反応しない場合どうするのか
- 近さだけでなく、営業時間・空き枠・安全性をどう推薦に反映するのか
- 荷物を預かった後、受取人とスポットはどう受取調整するのか
- 事故・紛失・期限超過・報酬保留をどう扱うのか
- 配送会社や店舗が使う理由は何か

v5 では、これらをプロダクトの中核要件として定義する。

## 1. プロダクトの結論

ShareKeep は、既存のコンビニ受取、宅配ボックス、置き配、PUDO を置き換えるものではない。

ShareKeep は、**それらで拾いきれない不在荷物を、地域の承認済み代理受取スポットへ一時退避させる補完インフラ**である。

### 1.1 一言説明

> ShareKeep は、不在時の荷物を地域の承認済み代理受取スポットへ一時保管し、受取人が帰宅後にQRで安全に受け取れる再配達削減サービスです。

### 1.2 避けるべき表現

| 避けたい表現 | 理由 | 置き換え表現 |
| --- | --- | --- |
| 近所の人に荷物を預ける | 不安・怪しさ・責任問題が強く見える | 承認済み代理受取スポットに一時保管する |
| 代理人マッチング | 個人間受け渡しアプリに見える | 代理受取スポット推薦 |
| 誰かに預かってもらう | 安全性が弱い | 審査済みスポットでQR保管する |
| チャットで受け渡し調整 | 運用が属人的に見える | 受取調整ステータスで管理する |

### 1.3 プロダクトの核

ShareKeep の価値は、以下の時間差を吸収することにある。

```text
配達員が届けられる時間  ≠  受取人が家にいる時間
```

このズレを、地域の店舗・施設・管理人室・審査済み個人などの代理受取スポットが持つ「空き時間」と「空きスペース」で吸収する。

## 2. 解く課題

### 2.1 社会課題

- 再配達により、配送会社の人件費・燃料費・業務負荷が増える
- 再配達により、車両走行が増え CO2 排出が増える
- 宅配ボックスがない住宅では、不在時の受取手段が限られる
- 置き配は盗難・雨濡れ・本人確認が必要な荷物に弱い
- コンビニ受取やロッカーは、事前指定していない荷物や空き枠不足に弱い

### 2.2 ユーザー課題

#### 受取人

- 日中に家にいないため、再配達依頼が面倒
- 夜なら近所の店舗・施設まで取りに行ける
- 知らない個人宅には預けたくない
- どこに荷物があるのか、いつ取りに行けるのかを明確に知りたい

#### 配達員 / 配送会社

- 不在時に持ち戻ると再配達が発生する
- 不在後の代替受取先がない
- 現場で長時間判断したくない
- 不正・誤配・責任問題を避けたい

#### 代理受取スポット

- 空き時間や空きスペースを活用したい
- 地域貢献や来店機会を作りたい
- ただし、無制限に荷物を預かりたくない
- トラブル時の責任や報酬条件を明確にしたい

## 3. 対象ロール

| ロール | UI上の名称 | DB上の既存値 | 主な役割 |
| --- | --- | --- | --- |
| 受取人 | 受取人 | `recipient` | 荷物登録、代理受取条件設定、候補選択、受取調整、QR提示、評価 |
| 代理受取スポット | 代理受取スポット | `agent` | スケジュール・保管枠登録、荷物保管、受取人QR読み取り、報酬受取 |
| 配達員 / 配送会社 | 配達員 / 配送会社 | `delivery_company` | 荷物管理、不在報告、スポットへの配送、スポットQR読み取り |
| 運営 | 運営管理者 | 将来追加 | スポット審査、トラブル対応、報酬保留解除、配送会社管理 |

### 3.1 重要な命名方針

既存実装との互換性のため、DB や内部コードで `agent` を使い続けてもよい。  
ただし、**画面・発表・ドキュメントでは原則「代理受取スポット」と呼ぶ**。

## 4. スコープ

### 4.1 ハッカソン MVP で見せる範囲

MVP では、外部 EC や配送会社 API との本格連携は行わず、アプリ内のデモデータで一連の体験を見せる。

必ず見せたい体験は以下。

1. 受取人が荷物を登録する
2. 受取人が代理受取条件を設定する
3. 配達員が不在報告をする
4. 推薦 API / 推薦ロジックが代理受取スポット候補を出す
5. 候補カードに、距離・受取可能時間・空き枠・推薦理由・除外理由が表示される
6. 受取人がスポットを選ぶ、または条件に合うスポットが仮割当される
7. 配達員が代理受取スポットへ配送し、スポットQRを読み取る
8. 荷物ステータスが「代理受取スポットで保管中」になる
9. 受取人が「19:30ごろ取りに行く」などの受取調整をする
10. 代理受取スポット側に到着予定が表示される
11. 受取人が受取用QRを提示し、スポットが読み取る
12. 完了画面で CO2 削減、配送会社請求イベント、スポット報酬イベントを表示する

### 4.2 MVP では実装しきらなくてよい範囲

- 本人確認書類の実アップロードと審査ワークフロー
- 実決済、振込、請求書発行
- 配送会社の実API連携
- ECサイトとの購入時連携
- 本番配送ルート最適化
- 事故補償の法務設計
- 高度な不正検知
- 本番データによる推薦モデル学習

これらは、将来構想として説明できればよい。

## 5. 基本フロー

### 5.1 ハッカソンデモフロー

```text
受取人
  1. 荷物を登録
  2. 代理受取条件を設定
     - 安全重視
     - 1km以内
     - 個人スポットNG
     - 19時以降に受け取りたい

配達員
  3. 配達開始
  4. 不在報告

推薦
  5. 候補を抽出
  6. ハードフィルタで不成立候補を除外
  7. 距離・時間・信頼度・空き枠でスコアリング
  8. 受取人に理由付きで提示

受取人
  9. みどり商店を選択

配達員 / 代理受取スポット
  10. 配達員がみどり商店へ配送
  11. みどり商店がスポットQRを表示
  12. 配達員がQRを読み取る
  13. 荷物が保管中になる

受取人 / 代理受取スポット
  14. 受取人が「19:30ごろ取りに行く」を押す
  15. スポット側に到着予定が表示される
  16. 受取人がQRを提示
  17. スポットがQRを読み取って引き渡し完了

完了
  18. CO2削減量を表示
  19. 配送会社請求イベントを表示
  20. スポット報酬イベントを表示
```

### 5.2 実運用フロー

本番では、受取人が毎回リアルタイムで候補を選ばなくても成立する必要がある。

```text
1. 受取人が事前に代理受取条件を設定する
2. 配送会社/ECから荷物情報が連携される
3. 配達前に候補スポットが事前計算される
4. 不在報告時に条件に合うスポットが自動提案される
5. 受取人が一定時間内に選択できる
6. 反応がない場合、事前条件に合う1位候補へ自動仮割当する
7. 候補なし・満枠・スポット停止中の場合は通常の再配達へ戻す
```

### 5.3 タイムアウト要件

受取人がすぐ反応しないと配達員が詰まるため、以下のタイムアウト設計を置く。

```text
不在報告後 0〜60秒:
  受取人が候補を選べる

60秒経過:
  事前条件に合う1位候補へ自動仮割当

候補なし / 満枠 / スポット拒否:
  通常の再配達へ戻す
```

MVP では、このタイムアウト処理を実装しきらなくてもよい。  
ただし、発表では「本番では毎回受取人の即時選択に依存しない」と説明する。

## 6. 代理受取スポット要件

### 6.1 スポット種別

代理受取スポットは以下に分類する。

| 種別 | 例 | MVPでの扱い |
| --- | --- | --- |
| 店舗 | 商店、カフェ、薬局、クリーニング店 | 最も推奨。デモの主役にする |
| 施設 | コワーキング、地域センター、大学施設 | 推奨 |
| 管理人室 | マンション管理室、寮管理室 | 推奨 |
| 個人 | 日中在宅者、近隣住民 | 受取人が許可した場合のみ候補化 |

### 6.2 スポット審査ステータス

```text
pending_review
approved
rejected
suspended
```

推薦対象になるのは原則 `approved` のみ。

### 6.3 スケジュール要件

現行の「曜日・開始時刻・終了時刻」だけでは、代理受取スポットとしては弱い。  
ShareKeep では以下を分けて扱う。

| 項目 | 意味 | 例 |
| --- | --- | --- |
| 基本対応曜日 | 通常営業日 | 月〜金 |
| 配達員受取可能時間 | 配達員から荷物を受け取れる時間 | 10:00〜18:00 |
| 受取人引き渡し可能時間 | 受取人へ荷物を渡せる時間 | 17:00〜21:00 |
| 当日受付締切 | 新規荷物を受け付ける締切 | 17:30 |
| 臨時休業 | 日付別の休み | 2026-06-15 は休み |
| 今日対応可能 | 当日だけ一時停止できる | ON / OFF |

重要なのは、**配達員から受け取れる時間** と **受取人へ渡せる時間** が別物であること。

例:

```text
みどり商店
- 配達員から受け取れる時間: 10:00〜18:00
- 受取人へ渡せる時間: 18:00〜21:00
```

これにより、日中に荷物を受け取り、夜に帰宅した受取人へ渡せる。

### 6.4 保管枠要件

スポットは無制限に荷物を預かれないため、保管枠を持つ。

```text
max_storage_count: 同時保管可能数
current_storage_count: 現在保管中の数
```

候補カードでは以下のように表示する。

```text
空き枠 2/5
```

MVP では日付別予約枠まで実装しなくてもよい。  
ただし、候補カードでは「空き枠あり」を見せる。

### 6.5 荷物制限

代理受取対象外にする荷物を明確にする。

```text
- 冷蔵・冷凍
- 生鮮食品
- 高額品
- 危険物
- 本人限定受取
- 着払い
- 大型荷物
- 破損リスクが高いもの
- 法令や配送会社規約上、代理受取不可の荷物
```

MVP では「通常小型荷物のみ対応」と画面に表示できればよい。

## 7. 受取人設定要件

### 7.1 事前設定

受取人はプロフィールまたは荷物登録時に、以下を設定できる。

```text
- 代理受取を利用するか
- 探索距離
- 個人スポットを許可するか
- 店舗/施設/管理人室のみ許可するか
- 受取希望時間帯
- 安全重視 / 近さ重視 / 夜間受取重視 / おまかせ
- お気に入りスポット
- NGスポット
- 荷物ごとの注意事項
```

### 7.2 テンプレート

設定項目が多いと使われにくいため、テンプレートを用意する。

| テンプレート | 条件 |
| --- | --- |
| 安全重視 | 店舗・施設・管理人室を優先。個人スポットは除外 |
| 近さ重視 | 1km以内を優先。距離スコアを強める |
| 夜に受け取りたい | 19時以降に受取可能なスポットを優先 |
| いつもの場所 | お気に入りスポットを優先 |
| おまかせ | 推薦APIに任せる |

MVP のデモでは、以下を使う。

```text
安全重視 / 1km以内 / 個人スポットNG / 19時以降受取希望
```

## 8. 推薦要件

### 8.1 基本思想

推薦は「一番近い人」ではない。  
推薦は、**安全に成立する候補を絞り、その中で距離・時間・信頼度・空き枠・配達員負担を総合評価する**。

処理順は以下。

```text
1. ハードフィルタで不成立候補を除外
2. 残った候補をスコアリング
3. 推薦理由と除外理由を生成
4. 受取人または配達員に候補を表示
5. 選択結果を推薦ログへ保存
```

### 8.2 ハードフィルタ

以下に該当する候補は出さない。

```text
- 審査済みスポットではない
- スポットが停止中
- 今日対応不可
- 探索距離外
- 配達員の許容迂回範囲外
- 受取人希望時間と合わない
- 保管枠が空いていない
- 荷物タイプが対象外
- 個人スポットで、受取人が個人利用を許可していない
- トラブル履歴が基準を超えている
- 評価が最低基準を下回る
```

### 8.3 スコアリング

| 要素 | 説明 | 推奨重み |
| --- | --- | ---: |
| 受取人からの距離 | 近いほど高い | 30% |
| 受取時間の一致 | 希望時間に合うほど高い | 20% |
| スポット信頼度 | 審査状態、評価、実績、トラブル履歴 | 20% |
| 保管枠 | 空きがあるほど高い | 10% |
| 配達員迂回距離 | 通常ルートから近いほど高い | 10% |
| スポット種別 | 店舗・施設・管理人室を少し優遇 | 5% |
| 受取人嗜好 | お気に入り/NG/過去利用 | 5% |

```text
recommendation_score =
  distance_score * 0.30 +
  schedule_score * 0.20 +
  trust_score * 0.20 +
  capacity_score * 0.10 +
  detour_score * 0.10 +
  spot_type_score * 0.05 +
  preference_score * 0.05
```

### 8.4 距離の考え方

遠すぎる候補は、たとえ安全でも受取人が取りに行かない。  
距離は以下のように扱う。

| 距離 | 表示・評価 |
| --- | --- |
| 0〜500m | 非常に良い |
| 500m〜1km | 良い |
| 1km〜2km | 条件次第で良い |
| 2km〜3km | 強い理由が必要 |
| 3km〜5km | 最終手段。遠め表示 |
| 5km超 | 原則除外 |

MVP では 1km 以内を基本にし、デモ都合で 1.2km の施設を 2位候補として出してもよい。

### 8.5 候補カード要件

候補カードには、スコアだけでなく、判断理由を表示する。

表示例:

```text
みどり商店
店舗スポット / 650m / 徒歩8分
本日 18:00〜21:00 受取OK
空き枠 2/5
おすすめ理由: 店舗スポット・夜まで受取可・高評価・空き枠あり
おすすめ度 92
```

### 8.6 除外理由

デモでは、除外理由を見せるとプロダクトの賢さが伝わる。

表示例:

```text
除外された候補
山田さん宅: 150m / 個人スポットNGのため除外
さくら管理人室: 400m / 18時閉室で希望時間に合わない
駅前ロッカー: 1.2km / 空き枠なし
```

## 9. QR・保管証跡要件

### 9.1 QRの役割

ShareKeep の安全性は QR を中心に作る。

| QR | 読み取る人 | 目的 |
| --- | --- | --- |
| 代理受取スポットQR | 配達員 | 荷物をスポットへ預けたことを確認する |
| 受取人QR | 代理受取スポット | 正しい受取人へ引き渡したことを確認する |

### 9.2 保管証跡

QRだけではなく、荷物の責任状態をイベントログとして残す。

新規テーブル案:

```text
parcel_custody_events
- id
- parcel_id
- event_type
  - assigned_to_spot
  - delivered_to_spot
  - stored
  - pickup_intent_created
  - handed_to_recipient
  - issue_reported
  - returned_to_carrier
- from_actor_id
- from_actor_type
- to_actor_id
- to_actor_type
- qr_verified
- photo_url
- location_lat
- location_lng
- note
- created_at
```

発表では以下のように説明する。

> ShareKeep では、荷物の責任状態を「配達員 → 代理受取スポット → 受取人」のイベントログとして残します。QRだけでなく、時刻・担当者・必要に応じた外装写真も記録するため、単なるチャット受け渡しではありません。

### 9.3 外装写真

MVP では必須ではない。  
ただし、将来は以下のタイミングで写真を残す。

```text
- 配達員がスポットへ預ける時
- スポットが保管開始する時
- 受取人へ引き渡す時
- トラブル報告時
```

## 10. 受取調整要件

### 10.1 基本思想

荷物を代理受取スポットが預かった後に必要なのは、自由入力チャットではなく、**受取調整ステータス**である。

```text
自由入力チャット = 補助
受取調整ボタン = 主導線
```

### 10.2 受取人側の操作

```text
- 今から取りに行く
- 10分後に着く
- 30分後に着く
- 19:30ごろ取りに行く
- 少し遅れる
- 本日中に行けない
- 場所がわからない
- 問題を報告する
```

### 10.3 代理受取スポット側の表示

```text
荷物カード
- 受取人名
- 荷物ID / 伝票番号
- 保管開始時刻
- 保管期限
- 受取人到着予定: 19:30ごろ
- 状態: pickup_scheduled
- 操作: QR読み取り / 連絡 / 問題報告
```

### 10.4 軽量テーブル案

MVP では `pickup_intents` を使う。

```text
pickup_intents
- id
- parcel_id
- recipient_id
- agent_id
- eta_minutes
- scheduled_at
- status: on_the_way / scheduled / delayed / canceled / arrived / completed
- message
- created_at
- updated_at
```

### 10.5 将来の予約枠テーブル

細かい受取時間枠を扱う場合は `parcel_pickup_reservations` を追加する。

```text
parcel_pickup_reservations
- id
- parcel_id
- recipient_id
- agent_id
- reserved_start_at
- reserved_end_at
- status: proposed / confirmed / canceled / expired / completed
- created_by: recipient / agent / system
- note
- created_at
- updated_at
```

```text
storage_started_at
storage_deadline_at
```

MVP では、以下のように扱う。

```text
保管期限: 当日23:59 または 24時間後
```

### 11.2 期限前通知

```text
- 保管開始時
- 期限2時間前
- 期限30分前
- 期限超過時
```

### 11.3 期限超過時

```text
- 受取人へ期限超過通知
- 代理受取スポットへ対応案内
- 配送会社へ回収依頼または再配達戻し
- 報酬を一時保留
```

MVP では、期限表示と「期限超過時は配送会社へ回収依頼」の説明だけでもよい。

## 13. お金・報酬・事業要件

### 13.1 基本方針

ShareKeep は、受取人課金を主軸にしない。  
受取人に「近くまで取りに行くための追加料金」を求めると使われにくい。

基本モデル:

```text
配送会社 / EC / 自治体
  ↓ 成功報酬・利用料
ShareKeep
  ↓ 報酬・ポイント・送客
代理受取スポット
```

### 13.2 配送会社への請求イベント

MVP では実決済せず、台帳イベントとして見せる。

```text
billing_events
- id
- parcel_id
- delivery_company_id
- recipient_id
- agent_id
- event_type: completed_via_sharekeep
- amount
- status: pending / confirmed / canceled / refunded
- created_at
- confirmed_at
```

### 13.3 スポット報酬イベント

```text
spot_reward_events
- id
- parcel_id
- agent_id
- reward_type: point / cash / coupon / store_promotion
- reward_amount
- status: pending / confirmed / held / canceled
- reason
- created_at
- confirmed_at
```

### 13.4 報酬確定条件

```text
- 配達員がスポットQRを読み取っている
- 荷物が代理受取スポットで保管中になっている
- 受取人QRで引き渡しが完了している
- 荷物ステータスが completed になっている
- 一定時間内にトラブル報告がない
```

### 13.5 採算説明

```text
ShareKeepの粗利 =
  配送会社/ECからの成功報酬
  - 代理受取スポットへの報酬
  - 問い合わせ/トラブル対応コスト
  - システム運用コスト
```

発表では、金額を断定しすぎず以下の言い方にする。

> 成功報酬は、配送会社が再配達1回を回避することで削減できる想定コストを上限に設定します。ShareKeep はその一部を代理受取スポットに還元します。

### 13.6 補助金の扱い

補助金は初期実証の加速手段であり、事業成立の前提にしない。

説明例:

> 地域実証や CO2 削減施策として補助金を活用できる可能性はありますが、事業の主軸は配送会社・EC事業者からの成功報酬です。

## 15. デモ設計

### 15.1 デモの狙い

デモでは、QRで荷物を渡すだけでは弱い。  
以下を見せる。

```text
- 受取人が条件を設定できる
- スポットが空きスケジュールと保管枠を持っている
- 推薦が最短距離だけではなく、安全性・時間・空き枠で判断している
- 荷物保管後に受取調整できる
- 完了後に CO2 削減、請求イベント、報酬イベントが発生する
```

### 15.2 デモ用 seed

| スポット | 距離 | 種別 | 条件 | 推薦結果 |
| --- | ---: | --- | --- | --- |
| 山田さん宅 | 150m | 個人 | 近いが受取人が個人NG | 除外 |
| みどり商店 | 650m | 店舗 | 21時まで、空き枠あり、高評価 | 1位 |
| さくら管理人室 | 400m | 管理人室 | 18時まで、希望時間に合わない | 除外または低順位 |
| コワーキングA | 1.2km | 施設 | 22時まで、高評価、少し遠い | 2位 |

### 15.3 デモシナリオ

```text
1. 受取人が荷物登録
2. 条件設定: 安全重視 / 1km以内 / 個人NG / 19時以降
3. 配達員が配達開始
4. 配達員が不在報告
5. 推薦候補が表示される
6. 山田さん宅は「個人NG」で除外される
7. みどり商店が「21時まで・空き枠あり」で1位になる
8. 受取人がみどり商店を選ぶ
9. 配達員がみどり商店へ配送
10. スポットQRで保管完了
11. 受取人が「19:30ごろ取りに行く」を押す
12. スポット側に到着予定が表示される
13. 受取人QRで引き渡し
14. 完了画面で CO2・請求イベント・報酬イベントを表示
```

### 15.4 審査員向け説明文

> この推薦は、単に最短距離の人を選んでいません。一番近い個人スポットは受取人が許可していないため除外され、夜まで営業していて空き枠のある店舗が選ばれます。荷物はQRでスポットに保管され、受取人は到着予定を共有したうえで、受取用QRで本人確認して受け取ります。完了時には、配送会社の再配達回避イベントとスポット報酬イベントが発生します。

## 16. 非機能要件

### 16.1 セキュリティ

- QRトークンは有効期限付き、使用済み管理あり
- QR検証は冪等にする
- 期待する荷物IDとQRの荷物IDが一致することを確認する
- RLS で、自分に関係する荷物・通知・メッセージのみ閲覧可能にする
- 代理受取スポットの詳細住所は確定前に出さない

### 16.2 監査性

- ステータス変更履歴を残す
- QR検証履歴を残す
- 保管証跡を残す
- トラブル報告履歴を残す
- 報酬・請求イベントを残す

### 16.3 可用性

- 推薦APIが落ちた場合、ルールベース推薦にフォールバックする
- 地図が表示できない環境ではリストで候補選択できる
- カメラが使えない場合、デモ用の手動遷移を用意してもよい

### 16.4 UX

- 候補カードでは「なぜおすすめなのか」を必ず表示する
- 受取人にとって遠すぎる候補は出しすぎない
- スポットには「現在の保管数」「今日対応可能ON/OFF」を目立たせる
- 自由入力より、定型ボタンで操作できるようにする

## 18. 想定質問と回答

### Q. コンビニ受取や PUDO でよくないですか？

ShareKeep はそれらを置き換えません。事前指定できていない荷物、空き枠がない荷物、不在後に行き場がなくなる荷物を、地域の承認済み代理受取スポットへ逃がす補完インフラです。

### Q. 個人に預けるのは危なくないですか？

個人スポットはデフォルトで優先しません。店舗・施設・管理人室を基本にし、個人スポットは本人確認・審査済み・受取人が許可した場合のみ候補に出します。QR、保管ログ、トラブル報告、報酬保留で安全性を担保します。

### Q. 配達員の負担が増えませんか？

推薦では受取人からの距離だけでなく、配達員の許容迂回距離も見ます。ルートから外れすぎる候補は除外し、成立しやすいスポットだけを提示します。

### Q. 受取人が取りに来なかったら？

保管期限を設定し、期限前に通知します。期限を超えた場合は、配送会社回収または再配達へ戻し、スポット報酬は一時保留にします。

### Q. AI推薦は本当に学習していますか？

MVP では合成データやルールベースを併用して構いません。重要なのは、距離・時間・信頼度・空き枠・受取人設定を特徴量として扱い、選択ログを保存して将来の再学習につなげる設計です。

### Q. どうやって収益化しますか？

受取人課金を主軸にせず、配送会社・ECからの成功報酬を基本にします。ShareKeep 利用料は再配達回避価値より低く設定し、その一部を代理受取スポットへ還元します。

---

## 19. 最終プロダクトストーリー

ShareKeep は、不在時の荷物をただ近所の誰かに預けるアプリではない。

受取人は、許容距離・受取時間・使ってよいスポット種別を設定する。  
代理受取スポットは、自分の空きスケジュールと保管枠を登録する。  
配達失敗時、推薦APIは距離・時間・安全性・保管枠・評価をもとに、成立しやすいスポットを選ぶ。  
荷物がスポットに届いた後、受取人は受取時間を共有し、QRで本人確認して受け取る。  
完了すると、配送会社には再配達回避の請求イベントが立ち、スポットには報酬イベントが立つ。

つまり ShareKeep は、

> 地域の空き時間・空きスペース・信頼できる受取スポットを使って、再配達の穴を埋めるラストワンマイル補完インフラ

である。
